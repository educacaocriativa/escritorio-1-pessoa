## Story
**As a** operador da plataforma,
**I want** um worker durável (cron/SQS) que processe o `tick` do funil e os envios assíncronos,
**so that** as esperas do funil e as notificações não dependam de chamada manual nem bloqueiem a request.

## Acceptance Criteria
1. Um worker/cron durável dispara `POST /funnels/runs/tick` periodicamente (hoje é endpoint manual/tela), retomando esperas vencidas de forma idempotente.
2. Os envios síncronos no request (ex.: WhatsApp ao mover card) migram para fila (SQS ou equivalente barato).
3. Auto-enroll por evento (lead criado/tag aplicada via `core/events`) é avaliado como próximo passo (documentado, não obrigatório nesta story).

### Integration Verification
- IV1: O motor de automação do funil (enroll→espera→tick→done) continua funcionando e idempotente.
- IV2: Falha de envio não derruba a request de origem.
- IV3: Custo do worker/fila respeita o guard-rail (NFR3).

### Technical Constraints
- **Sem dependência nova** (nem `boto3`, nem `apscheduler`/`celery`) — loop `time.sleep` da stdlib, decisão justificada na Task 3 (AUTO-DECISION) por custo (NFR3) e por a produção real ser VPS Docker Compose, não AWS.
- **RLS respeitada:** o worker NUNCA usa uma sessão sem tenant para ler/escrever tabelas de negócio (`FunnelRun`, `Notification`) — sempre abre `tenant_session(tenant_id)` por tenant, um de cada vez, exatamente como uma request autenticada faria. Só a listagem de `Tenant` (tabela global) usa sessão sem tenant.
- **Idempotência (IV1):** garantida por `engine.tick()` já existente (não alterado) + pelo fato de `process_pending` só reprocessar linhas com `status=="pending"` (uma vez enviada, a notificação sai do conjunto processável — não há reenvio automático/retry nesta story; `attempts`/`last_error` ficam registrados para uma dívida futura de retry-with-backoff, fora do escopo literal das ACs).
- **Concorrência:** esta story assume UMA única réplica do container `worker` (não há `FOR UPDATE SKIP LOCKED` nem lock distribuído). Se no futuro o worker escalar para múltiplas réplicas, will precisar de um lock (ex.: `SELECT ... FOR UPDATE SKIP LOCKED` no `process_pending`, mesmo padrão já usado em `payout`/`settle` da Carteira para lock otimista) — documentar como dívida, não implementar agora (não pedido pela AC, adicionaria complexidade sem necessidade no volume atual do produto).
- **Import de `Tenant`/`PLATFORM_SLUG`:** `app.worker` importa de `app.modules.auth.models` (Tenant) e `app.seed` (PLATFORM_SLUG) — sem risco de import circular (nenhum dos dois importa `app.worker`).
- **Gap de documentação sinalizado (não resolvido nesta story):** `CLAUDE.md` §2 lista "Infra AWS: container enxuto (EC2 Graviton + Docker) → ECS Fargate" como a stack decidida, mas `docs/HOSTINGER-DEPLOY.md` confirma que a produção REAL hoje é uma VPS Hostinger com Traefik compartilhado. As duas fontes coexistem como "AWS Fase A" vs. "VPS (opção B)" sem uma nota explícita de qual é o alvo ATUAL de produção — a Task 3 decidiu pelo alvo VPS por ser o comprovadamente em uso (`e1p.doroeventos.com.br`), mas esse gap de clareza no `CLAUDE.md` fica sinalizado como risco de leitura para as próximas stories de infra do Epic 4 (ex.: Story 4.4, wildcard de subdomínio, que já assume Caddy/DNS-01 — mesma direção "VPS" desta decisão).

### Testing
- Framework backend: `pytest` (`apps/api/tests`), mesmo padrão dos demais módulos (fixtures de `conftest.py`, SQLite em memória, `monkeypatch` para `whatsapp.send_text`/`email.send_email`).
- **Padrão de injeção de sessão obrigatório em `run_sweep`:** para ser testável sem Postgres real, `run_sweep()` deve aceitar `session_factory`/`tenant_session_factory` como parâmetros com defaults para os reais (`SessionLocal`/`tenant_session`), permitindo que `test_worker.py` injete factories que apontam para a sessão SQLite compartilhada — mesmo idioma de `get_tenant_session_factory` + `conftest.py::_override_factory`.
- Testes novos obrigatórios: ver Task 6 (`test_notifications_queue.py`, ajuste em `test_notifications.py`, `test_worker.py` completo).
- **Limitação conhecida:** não há Docker Compose completo (Postgres+API+worker) disponível neste sandbox de draft para validar o container `worker` de ponta a ponta — a validação de subir o serviço de fato (`docker compose up -d worker`, ver logs) fica registrada como validação manual pós-implementação (Task 7), mesma categoria de limitação já registrada em outras stories deste projeto para fluxos sem ambiente completo disponível.
