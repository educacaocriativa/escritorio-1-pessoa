## Story
**As a** dono do negócio,
**I want** mover os anexos de bytes-no-Postgres para object storage isolado por tenant,
**so that** o banco/backup não inche e os arquivos escalem com segurança.

## Acceptance Criteria
1. O módulo de Anexos passa a persistir os arquivos em object storage S3-compatível (mantendo `owner_type`/`owner_id`/`label` e o isolamento por tenant), em vez de `LargeBinary` no Postgres.
2. Upload, listagem, download e delete continuam funcionando pela mesma interface (componente `Attachments.tsx` e endpoints atuais inalterados no contrato).
3. Anexos existentes são migrados (ou plano de migração documentado); o download de anexos antigos continua funcionando.

### Integration Verification
- IV1: Contas a Pagar/Receber (boleto/contrato) e a Agenda (anexos do evento) continuam exibindo/baixando os arquivos.
- IV2: Isolamento por tenant preservado (um tenant não acessa arquivo de outro).
- IV3: O tamanho do dump do Postgres cai após a migração (validar no backup).

### Technical Constraints
- Não remover a coluna `data` da tabela `attachments` nesta story (mantém compatibilidade de leitura para anexos não migrados — AC3). Removê-la é uma limpeza futura, só depois que o backfill (Task 5) for confirmado 100% em produção.
- Não introduzir chamada de rede (S3) dentro da própria migration Alembic (`0039`) — migrations rodam automaticamente no boot do container (`alembic upgrade head`, conforme `CLAUDE.md`/epic); acoplar I/O de rede/credenciais a esse caminho arriscaria quebrar o boot em qualquer ambiente sem bucket configurado. A migration é estritamente estrutural; o backfill é um script operacional separado (Task 5), rodado manualmente/documentado.
- Preservar exatamente o contrato HTTP dos 4 endpoints de `/attachments` (path, verbos, request/response shape) — AC2 é explícito quanto a isso.
- `storage_key` nunca deve vazer para o frontend/API pública — é detalhe de implementação, mantém `AttachmentOut` como está.
- Seguir a Regra de Ouro nº 1 (isolamento de tenant) também no path do object storage (`build_key` prefixado por `tenant_id`), mesmo que a autoridade real de acesso continue sendo a autenticação/RLS do metadado — defesa-em-profundidade específica para o novo componente (storage), não uma mudança na política geral de "RLS como única garantia" já decidida para o banco (ver `CLAUDE.md` nota de arquitetura mantida).
